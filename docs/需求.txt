总目标：将大型 PDF 文件转换为 Markdown 文件

本文档是本仓库的“总需求概览”和“需求入口”，用于统一说明四个阶段的总体目标、阶段边界和需求文档索引。

说明：
1、Phase 级别的正式需求、非功能要求和验收标准，以对应 PRD 为准。
2、技术方案和开发计划应分别引用各自对应的 PRD，不再默认仅引用 Phase 1 PRD。
3、本文档保留最初需求草案的核心意思，但不再承担各阶段详细需求冻结职责。

一、总体处理链路
1、Phase 1：将源 PDF 按章节、页数与语义完整性要求切分为多个 PDF 切片，并输出 `manifest.json`
2、Phase 2：对 Phase 1 的切片 PDF 执行内容提取，输出 `content.json`、资源文件、`extract_manifest.json`，并可选输出同名 Markdown 草稿供人工检查
3、Phase 3：以 Phase 2 的 `content.json` 为事实基线进行完整性校对与格式修复，输出最终版 Markdown、`review_report.json` 与 `format_manifest.json`；若存在 draft，则仅作为辅助对照信息
4、Phase 4：对 Phase 3 的最终版 Markdown 执行全局校验、overlap 去重与合并，输出最终单文件 Markdown、`merge_report.json` 与 `merge_manifest.json`

二、阶段需求文档索引
1、Phase 1 PRD：`docs/PRD-PDF切分需求.md`
2、Phase 2 PRD：`docs/PRD-PDF内容提取需求(Phase2).md`
3、Phase 3 PRD：`docs/PRD-Markdown格式化需求(Phase3).md`
4、Phase 4 PRD：`docs/PRD-校验与拼接需求(Phase4).md`

三、阶段边界摘要
1、Phase 1 负责 PDF 结构化切分，不负责 Markdown 提取与格式化
2、Phase 2 负责内容提取、文档树建模和追溯信息保留，核心契约是 `content.json` / `extract_manifest.json` / `assets/`，不负责最终精排
3、Phase 3 负责基于 `content.json` 的完整性对账与 Markdown 标准化，不负责重新抽取 PDF 内容，也不负责 overlap 去重
4、Phase 4 负责全局校验、相邻切片 overlap 去重、资源路径收敛与最终合并，不负责自由改写正文

四、原始需求摘要（历史草案，仅用于追溯，不作为当前冻结契约）

说明：
1、以下内容保留项目最初的口头/草案需求表述，便于追溯来源。
2、若与“总体处理链路”“阶段边界摘要”或各 Phase PRD 冲突，以 PRD 和技术方案中的现行契约为准。
3、尤其是 Phase 2/3 中关于 “MD 文件是否为核心输入输出” 的描述，应按现行文档理解为：Phase 2 以 `content.json` / `extract_manifest.json` / `assets/` 为核心契约，Phase 3 以 `content.json` 为事实基线，draft `.md` 仅为可选辅助产物。

phase1：对于大型 PDF 文件转 MD 文件，第一步需要对 PDF 文件进行切分，核心要求如下：
1、按章节切分，不要把一章节的内容切成多份
2、页数控制在 20 页左右，如果一章的内容超过 20 页，可以切成多个小节，但是要保证每一份都是完整的章节
3、如果一章的内容少于 20 页，可以和下一章的内容合并，但是要保证合并后的内容不超过 20 页
4、尤其要确保一个段落不会被切成两半、表格不会被切成两半、代码不会被切成两半
5、对于章节之间的切换页面，要确保章节名和章节内容都在同一个文件中，前后章节都要保留这一页内容，不要遗漏，避免章节内容不完整
6、切分后的文件命名为：章节名（页码范围）.pdf

phase2：对 Phase 1 切分后的 PDF 文件进行内容提取，核心要求如下：
1、严格按照 PDF 的格式进行提取，不要遗漏任何内容
2、提取后的内容保存为 MD 文件，命名为：同名文件.md

phase3：对 Phase 2 提取后的 MD 文件进行校对检查，确保内容完整，没有遗漏：
1、逐个章节（文件）检查 MD 文件内容是否完整，没有遗漏
2、逐个章节（文件）检查 MD 文件格式是否正确，没有遗漏

phase4：对 Phase 3 校对后的 MD 文件进行合并，合并规则如下：
1、将所有 MD 文件按照章节顺序合并
2、合并后的文件命名为：原文件名.md
