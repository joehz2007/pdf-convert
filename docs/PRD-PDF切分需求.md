## 1. 产品概述

### 1.1 项目背景
在将包含数百以至上千页的大型 PDF 文件转换为 Markdown (MD) 格式的过程中，受限于大模型处理上下文 token 的限制以及转换的准确度要求，需要对源 PDF 文件进行结构化、合理化的切分处理。

### 1.2 目标定义
本项目聚焦于 **PDF 转换为 MD 文件的第一阶段：PDF 切分**。

其核心目标是：实现一个 **Python CLI 工具**，面向**数字原生或具备可提取文本层的 PDF 文档**，自动识别 PDF 文件内在结构，并结合文本块几何信息与轻量版面分析能力，在保证内容语义完整性的前提下，将其切分为多个以 20 页为目标、25 页为常规弹性上限的 **独立 PDF 子文件**，供后续 Phase 2 内容提取模块使用。

> **关键约定**：Phase 1 的职责包括物理切分、结构识别和基于文本层/几何信息的切分边界判断；**本期不引入 OCR**。其输出物为切分后的 **PDF 文件** 和结构化清单 **manifest.json**。MD 转换属于 Phase 2/3 的工作范围。
>
> **Phase 职责边界**：Phase 1 仅利用 PDF 现有文本层和版面几何信息做结构识别与边界判断；Phase 2 若需处理扫描版 PDF 或进行全文文字提取，由 Phase 2 单独负责 OCR 管线建设与内容抽取。

### 1.3 适用场景
- 处理动辄几百页的书籍、报告、长篇说明书等大型 PDF 资料。
- 确保切分后的片段适合后续 LLM (大型语言模型) 或文本提取/OCR 引擎进行高精度的批量文本提取和 Markdown 格式化。

## 2. 系统阶段规划 (Context)

本节定义的阶段划分**替代原始需求草案中的阶段划分**，后续技术方案、任务拆解、测试计划与排期统一以本 PRD 为准。系统总体处理流程规划如下：

- **Phase 1: PDF 结构化切分模块（当前范围）**：面向数字原生或具备可提取文本层的 PDF，根据章节、页数和内容完整性规则，结合结构识别与版面几何分析，将源大文件切分为多个 PDF 子文件，并输出切分清单 manifest.json。
- **Phase 2: PDF 内容提取模块**：基于切分后的 PDF 片段，进行 OCR 或文本解析操作；若需支持扫描版 PDF 的全文抽取，由本阶段负责 OCR 管线。
- **Phase 3: Markdown 格式化模块**：将解析结果排版为带样式的标准 MD 格式。
- **Phase 4: 校验与拼接模块**：内容核对与多文档合并（可选）。

## 3. 功能需求（核心切分规则）

系统的核心逻辑是一个基于规则的 **切分决策引擎**。具体规则定义如下：

### 3.1 规则优先级决策表

当多条规则发生冲突时，系统必须按照以下优先级进行裁决：

| 优先级 | 规则 | 类型 | 说明 |
| :--- | :--- | :--- | :--- |
| **P0** | 文件必须成功生成 | 硬约束 | 系统不得因规则冲突进入死循环、崩溃或产出空结果 |
| **P1** | 语义块完整性（F3） | 硬约束 | 段落、表格、代码块、图片与图注组合不得被切断 |
| **P2** | 章节/小节完整性（F1/F2） | 强约束 | 优先保持顶层章节完整；必要时下钻至二级小节；二级小节优先保持完整 |
| **P3** | 过渡页双向冗余（F4） | 强约束 | 章节切换页应同时出现在前后相邻文件中 |
| **P4** | 目标页数 20 页 | 软约束 | 作为优先追求的默认目标 |
| **P5** | 常规弹性上限 25 页 | 强约束 | 在不破坏 P1-P3 的前提下，单文件通常不应超过 25 页 |
| **P6** | 超限异常输出 | 兜底规则 | 若单一完整小节或单一不可分割语义块自身超过 25 页，允许输出超限切片，但必须记录异常并标记人工复核 |

补充决策规则：

- 若 `20 页目标` 与 `语义块完整性` 冲突，以语义块完整性优先。
- 若 `25 页常规上限` 与 `完整小节` 或 `不可分割语义块` 冲突，允许生成超限切片，不强制在语义块内部截断。
- 任何超限切片都必须在 `manifest.json` 中记录 `exception_type`、`actual_pages`、`boundary_reason`，并设置 `manual_review_required=true`。
- “最近安全边界”定义为：不破坏段落、表格、代码块、图片与图注组合完整性的最近切分位置。

### 功能1：基于顶层章节结构切分 (F1 - Chapter Based Slicing)

- **章节层级基准**：以 PDF 文档中的**顶层章节**（即 TOC 或书签中的第一级条目，如"第一章"、"Chapter 1"）为基础切分单元。
  - 在满足页数约束（F2）的前提下，优先保持顶层章节的完整性，不将其拆散到多个文件。
  - 若顶层章节内容过长超过阈值，则向下进入二级标题（Section）维度进行切分，以保证每个子节的独立性和完整性。
- **章节识别优先级**（见 F 降级，第4节）：优先使用 TOC/书签，其次使用章节编号正则 + 字体样式启发式识别，并结合文本块几何信息辅助判断。
- **依赖条件**：系统需要具备读取和解析 PDF 目录（TOC）、书签（Bookmarks）、文本层和版面几何结构的能力。

### 功能2：页数阈值控制 (F2 - Page Limit Control)

- **目标阈值**：单个切分文件的页数应控制在 **20页以内**。
- **常规弹性上限**：当 F3 语义完整性约束触发时，允许页数放宽至 **最多 25 页**。
- **超限异常**：若单一不可分割语义块或单一完整小节自身已超过 25 页，则允许生成 **超限切片**，但必须在运行日志和 `manifest.json` 中输出警告，并标记人工复核；此场景下**不允许**为了回到 25 页以内而在语义块内部强制切断。
- **详细策略与边界逻辑**：
  1. **规则 2.1 (单章超限)**：如果单个顶层章节内容总页数 **> 20 页**：
     - *策略*：允许打破"不拆分章节"的规则，将其切分为多个更小的"小节（Sub-sections）"。
     - *小节识别方式*：**优先按文档内的章节编号**（如 1.1、1.2、2.1 等带编号的二级标题）识别小节边界；若无编号，则退而使用字体字号启发式识别（字号较大、加粗等）判断标题行；若仍无法识别，按物理段落边界在最近安全位置切分。
     - *约束*：底层切分单元应优先保持为**完整的小节**。若单一完整小节本身已超过 25 页，则允许其以“超限切片”方式单独输出，并进入人工复核。
  2. **规则 2.2 (单章过短与合并)**：如果单个顶层章节内容总页数 **< 20 页**：
     - *策略*：**保守合并**——依次尝试将当前章节与紧邻的下一个章节合并。若合并后总页数 ≤ 20 页则合并，否则当前章节独立成文件，不继续尝试。
     - *关键约束*：合并后的总页数**绝对不允许**超过 20 页（语义完整性弹性除外）。
     - *示例*：章A(10页) + 章B(8页) = 18页 → 可合；章A(10页) + 章B(12页) = 22页 → 不可合，章A独立。

### 功能3：语义块完整性约束 (F3 - Semantic Block Integrity)

- **优先级**：此规则具有**最高优先级**，遇到与页数阈值冲突时，允许页数在弹性上限（25页）范围内浮动；**若 25 页仍无法满足语义完整性，按 3.1 规则优先级决策表的 P6 处理**。
- **规则定义**：在决定确切的切分页面边界时，**绝对禁止**切断以下核心语义块的连续性：
  - 断开一个完整的**文字段落**。
  - 断开一个跨页的**数据表格**。
  - 断开一个跨页的**代码块**。
  - 断开一个由**图片与其紧邻图注/说明文字**构成的复合语义块。
- **识别手段**：语义块识别可综合使用文本层解析、块级几何位置信息与轻量版面分析结果。

### 功能4：平滑过渡页容错 (F4 - Transitional Context Splicing)

- **"切换页"的定义**：指章节 A 的末尾内容与章节 B 的起始内容（含章节标题）共同出现的那一个物理页面。若章节切换发生在同一页（上半页为 A 章结尾、下半页为 B 章开头），则该整页即为切换页。
- **处理策略（冗余重叠）**：生成切分文件时，必须确保前后相邻的两个文件都**全量保留**该切换页的完整内容（即该页同时出现在前一个文件的末尾和后一个文件的开头）。
- **目的**：保证章节名和章节上下文在同一个文件中同时出现，不产生结构断层或内容截断。

### 功能5：规范化文件输出 (F5 - Normalized Output)

- **输出格式**：切分产物为 **PDF 文件**。
- **命名规范**：统一采用 `<章节名>（<起始页>-<结束页>）.pdf` 的格式命名。
  - 示例：`第一章 系统概述（1-18）.pdf`、`Chapter 2 Architecture（19-35）.pdf`
  - 页码以源文件的物理页码（从第1页起）为准。
- **文件名合法性处理**：
  - 对于章节名中包含非法文件名字符（如 `\ / : * ? " < > |`），自动替换为全角字符或下划线 `_`。
  - 若存在重名现象（如多个同名章节），追加序号区分（如 `附录（210-215）_01.pdf`）。
- **输出目录**：切分后的文件统一输出至源文件所在目录下的 `<源文件名>_split/` 子目录中。
- **结构化清单输出**：除 PDF 切片外，系统必须额外输出一个 `manifest.json` 文件，位于同一输出目录，用于为后续 Phase 2-4 提供稳定输入。
  - `manifest.json` 采用“全局元数据 + 切片数组”的层级结构，推荐骨架如下：

```json
{
  "source_file": "input.pdf",
  "total_pages": 500,
  "created_at": "2026-03-21T10:00:00Z",
  "generator_version": "phase1-v1",
  "fallback_level": 1,
  "slices": [
    {
      "slice_file": "第一章 概述（1-18）.pdf",
      "start_page": 1,
      "end_page": 18,
      "actual_pages": 18,
      "display_title": "第一章 概述",
      "toc_level": 1,
      "split_mode": "chapter",
      "overlap_pages": [],
      "boundary_reason": "chapter_boundary",
      "exception_type": null,
      "manual_review_required": false
    }
  ]
}
```

  - 全局字段最低必填包括：
    - `source_file`
    - `total_pages`
    - `created_at`
    - `generator_version`
    - `fallback_level`
    - `slices`
  - 切片字段最低必填包括：
    - `slice_file`
    - `start_page`
    - `end_page`
    - `actual_pages`
    - `display_title`
    - `toc_level`
    - `split_mode`
    - `overlap_pages`
    - `boundary_reason`
    - `exception_type`
    - `manual_review_required`
  - 字段枚举约定：
    - `split_mode`：`chapter` / `section` / `merge` / `physical`
    - `boundary_reason`：`chapter_boundary` / `section_boundary` / `page_limit` / `semantic_integrity` / `fallback_physical`
    - `exception_type`：`oversized_section` / `oversized_semantic_block` / `null`
    - `fallback_level`：`1` / `2` / `3`
    - `overlap_pages`：页码数组，例如 `[]`、`[15]`、`[15, 16]`
  - 页码口径约定：`manifest.json`、输出文件名、日志中对用户可见的页码均采用**1-based 物理页码**。

## 4. 非功能性需求（NFRs）

### 性能要求
- 性能指标采用**分模式验收**，不同识别模式分别定义：
  - **Level 1（TOC/书签直切）**：处理 1000 页以内的数字原生 PDF，`预检 + 结构识别 + 切分写盘` 总耗时应控制在 **30 秒以内**。
  - **Level 2（文本层 + 版面分析）**：处理 1000 页以内、可提取文本但无 TOC/书签的 PDF，`预检 + 标题识别 + 版面分析 + 切分写盘` 总耗时应控制在 **60 秒以内**。
- **当前版本不对扫描版 PDF 定义性能指标**，因为扫描件 OCR 与重建识别能力不在本期范围内。
- 标准测试环境建议定义为：**8 核 CPU、16GB RAM、SSD、本地文件系统、Windows 11 或同等级 Linux 环境**。
- 性能计时范围：从读取源 PDF 到完成所有切片文件、`manifest.json` 写入磁盘为止。
- 系统日志必须输出分阶段耗时，至少包括：`precheck_ms`、`structure_detect_ms`、`layout_analysis_ms`、`split_write_ms`、`total_ms`。

### 容错性：三级降级策略（Fallback）
当 PDF 缺乏结构化信息时，系统按以下优先级依次降级，并在运行日志中明确标注当前使用的识别模式：

| 级别 | 触发条件 | 处理策略 |
| :--- | :--- | :--- |
| **Level 1（正常）** | PDF 包含有效的 TOC 或书签（Bookmarks） | 直接基于 TOC/书签层级结构切分 |
| **Level 2（降级）** | 无 TOC/书签，但 PDF 可提取文本内容 | 基于章节编号正则（如 `第\d+章`、`^\d+\.\s`）+ 字体字号启发式 + 版面结构特征识别章节边界 |
| **Level 3（不支持输入）** | 无结构信息，或仅有扫描图像，无法直接提取有效文本层 | 当前版本判定为**超出本期范围**，终止处理并输出明确错误信息；不得产生部分切片文件。日志中记录 `fallback_level=3`，并提示转交后续 OCR 方案处理 |

### 鲁棒性
- 加密 PDF：预检阶段识别并提前提示用户，终止处理，不产生部分输出。
- 损坏 PDF：异常捕获后输出错误信息，说明失败原因，不崩溃退出。
- 空文件 / 0页 PDF：前置校验拦截，输出明确错误信息。

### 部署形态
- **Python CLI 脚本**，直接在命令行中执行。
- 基础调用方式：`python split_pdf.py <input.pdf> [--output-dir <dir>] [--max-pages 20]`

## 5. 验收标准

1. **结构识别**：输入一份包含明确书签、总页数 > 100 页的测试 PDF，系统能自动产出多个以所在章节命名的 PDF 切片文件，文件名包含正确的页码范围。
2. **页数控制**：抽查任意生成的单个文件：
   - 正常情况下，确认页数 ≤ 20 页。
   - 因 F3 触发弹性浮动的文件，确认页数 ≤ 25 页，且运行日志中有对应告警或说明记录。
   - 若因单一完整小节或单一不可分割语义块自身超限而产生 >25 页切片，确认 `manifest.json` 中包含异常标记与人工复核标记。
3. **语义完整性**：人工检查所有切分边界（上一个文件的尾部和下一个文件的头部），确认未出现被截断的段落、断开的表格、断开的代码块，以及被拆散的图片与图注组合。
4. **过渡页冗余**：明确检查章节边界页面，确认相邻的切分文件都成功冗余包含了该切换页的完整内容。
5. **文件命名**：所有文件名合法，不含非法字符，章节名与片段内容对应，页码范围准确。
6. **结构化清单**：输出目录中存在 `manifest.json`，且字段完整、页码映射准确、重叠页与异常标记可被后续模块消费。
7. **不支持输入处理**：输入一份无书签且无可提取文本层的扫描版 PDF，系统不崩溃，能够明确终止处理并输出“当前版本不支持扫描版 PDF”的错误信息，不产生部分输出。
8. **CLI 可用性**：`--help` 输出参数说明；非法参数或不存在的文件路径有清晰的错误提示。
9. **性能验收**：在约定测试环境下，Level 1、Level 2 分别满足第 4 节定义的性能目标，并可在日志中看到各阶段耗时统计。
